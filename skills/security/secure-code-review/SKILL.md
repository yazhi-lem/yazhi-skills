name: secure-code-review
description: Use when reviewing a pull request or diff that touches authentication, data input handling, deserialization, or dependencies — before merge, not after. Helps you systematically catch injection, broken authorization, and unsafe dependency changes that functional review tends to miss.
---

Security-focused code review is a distinct pass from functional/style review — it assumes every input is hostile and every boundary is a place authorization can be forgotten. Run it on any diff touching input parsing, auth/authz, serialization, external calls, or third-party dependencies, before merge.

## Workflow

1. **Identify every external input in the diff.** User-supplied fields, query params, headers, file uploads, webhook payloads, message queue contents. Anything not generated by your own trusted code is external input.
2. **Trace each input to its sink.** Does it reach a SQL query, shell command, file path, template renderer, deserializer, or another service's API? Each sink type has a matching injection class — check the table below.
3. **Verify authorization at every boundary the diff touches**, not just the outermost endpoint. Check object-level authorization (can user A access user B's resource by ID?) in addition to route-level auth. This is the single most commonly missed class of bug.
4. **Check deserialization paths.** Is any user-controlled data passed to a deserializer (pickle, YAML full-load, Java native serialization, PHP unserialize) that can construct arbitrary objects? Prefer schema-validated formats (JSON with a schema, protobuf) over generic deserializers.
5. **Review dependency changes.** Any new or bumped dependency: check for known CVEs, maintenance status, and whether the added transitive tree is proportionate to the feature. A one-line utility shouldn't pull in a network-capable package.
6. **Check error handling and logging.** Do error paths leak stack traces, internal paths, or secrets to the client or logs? Are secrets ever interpolated into log lines?
7. **Confirm output encoding matches the sink.** HTML-escape for HTML context, parameterize for SQL, shell-escape (or better, avoid shell entirely) for commands.
8. **Score findings and gate merge** using the table below.

## Injection class checklist

| Class | Where to look | Safe pattern |
|---|---|---|
| SQL injection | Any string-built query | Parameterized queries / prepared statements only |
| Command injection | `exec`, `system`, shell-outs with user input | Avoid shell; use argument arrays, allowlist inputs |
| XSS | Template rendering, `innerHTML`, dangerouslySetInnerHTML | Context-aware output encoding, CSP, avoid raw HTML sinks |
| Path traversal | File paths built from user input | Canonicalize + allowlist base directory, reject `..` |
| Deserialization | pickle/YAML/native (un)serialize on user data | Schema-validated formats, safe-load only |
| SSRF | Server making outbound requests to user-supplied URLs | Allowlist destinations, block internal IP ranges |
| Broken object-level authz | Any `GET/PUT/DELETE /resource/:id` handler | Check ownership/permission for every ID, every time |

## Severity gate

| Severity | Example | Action |
|---|---|---|
| Critical | Unauthenticated RCE, auth bypass, SQLi on prod data | Block merge, fix immediately |
| High | Missing object-level authz, stored XSS, known-exploited CVE in dependency | Block merge |
| Medium | Reflected XSS needing user interaction, verbose error leakage | Fix before next release |
| Low | Missing security headers, outdated but unexploited dependency | Track as backlog, fix within 30 days |

## Anti-patterns

- **Reviewing only the lines changed, not the sink they feed.** A one-line diff can introduce injection three functions downstream; trace the data flow, not just the diff hunk.
- **Trusting route-level auth as sufficient.** Object-level authorization checks are the most frequently skipped control — verify them per resource access, not just per endpoint.
- **Approving dependency bumps without checking changelogs/CVEs.** A transitive dependency bump can introduce a vulnerable version silently.
- **Rejecting deserialization risk as "theoretical."** Deserialization RCEs are routinely exploited in the wild; treat unsafe deserializers of user data as Critical by default.
- **Letting verbose errors ship "for debugging."** Stack traces and internal paths in prod responses are a reconnaissance gift to attackers.

Design-level issues (missing trust boundary, wrong architecture) should go back to `threat-modeling` rather than being patched at the code level. LLM/agent-specific code paths need `llm-security-review` in addition to this. Any hardcoded credentials found during review go through `secrets-management`, and a suspected active exploit found in review triggers `incident-response`.
